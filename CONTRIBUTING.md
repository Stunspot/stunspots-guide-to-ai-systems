# Contributing

Thanks for your interest in improving *Stunspot’s Guide to AI Systems*.

This repository is primarily a curated public knowledge canon, not an open collaborative textbook. Suggestions, corrections, source-quality notes, typo fixes, broken-link reports, and clarity improvements are welcome. Large unsolicited rewrites or structural changes may not be accepted.

## Good Contributions

Useful contributions include:

- typo, formatting, or broken-link fixes;
- corrections to factual claims;
- better citations or source-quality notes;
- reports of duplicated, stale, or contradictory material;
- accessibility or navigation improvements;
- small wording changes that improve clarity without changing doctrine.

## Before Opening an Issue or Pull Request

Please check whether the issue affects:

1. a canonical report in `/docs/`;
2. a bundled knowledge pack in `/knowledge-packs/`; or
3. repository metadata such as README, citation, license, or site navigation.

The canonical report files in `/docs/` are the source of truth. Knowledge packs are bundled derivatives and may be regenerated after source edits.

## Reporting Problems

When reporting an issue, include:

- the affected file path;
- the section heading or nearby text;
- what appears wrong or unclear;
- any suggested correction;
- supporting source links, if relevant.

For factual corrections, please distinguish between:

- confirmed error;
- outdated claim;
- source disagreement;
- unclear phrasing;
- personal preference.

That distinction saves everyone from playing epistemic whack-a-mole with a fog machine running.

## Pull Request Guidelines

Keep pull requests small and focused.

A good pull request usually changes one thing:

- one typo sweep;
- one broken-link repair;
- one citation improvement;
- one small wording clarification;
- one metadata or navigation fix.

Avoid bundling unrelated edits into a single PR.

## Style Notes

Please preserve the canon’s tone: clear, practical, doctrine-oriented, and model-usable. This repository is written for human readers, but it is also designed as AI/RAG substrate. Edits should improve retrieval clarity, conceptual precision, and downstream usability.

Prefer:

- specific terms over vague generalities;
- short section headings with strong semantic labels;
- explicit source boundaries;
- practical distinctions over ornamental theory;
- clear Markdown that remains useful when ingested by AI tools.

Avoid:

- generic AI hype language;
- unexplained jargon;
- flattening sharp distinctions into mush;
- marketing filler;
- large tone shifts;
- changes that make the text prettier but less usable as a knowledge substrate.
